# E2E 端到端测试（Playwright）

## 什么是 E2E 测试？

类比：**用户体验质检员**——不是检查零件是否合格（那是单元测试的工作），而是模拟真实用户从头到尾走一遍完整流程，确保整个系统能正常运转。

E2E（End-to-End）测试在真实浏览器中模拟用户操作，验证从前端页面到后端 API 的完整链路。本项目的 E2E 测试覆盖了页面加载、用户认证、搜索、导航、API 接口、响应式布局、WebSocket 连接、AI 聊天以及历史 Bug 回归等关键场景。

```
用户操作 → [浏览器] → [前端页面] → [HTTP/WebSocket] → [后端 API] → [数据库]
   ↑                                                                    |
   └────────────── E2E 测试验证整条链路 ──────────────────────────────────┘
```

## 测试框架与配置

测试基于 **Playwright** 实现，配置文件为项目根目录的 `playwright.config.js`。

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 测试目录 | `./e2e` | 所有测试文件存放位置 |
| 并行执行 | `fullyParallel: true` | 测试用例并行运行，加快执行速度 |
| CI 重试 | `retries: 2` | CI 环境失败后自动重试 2 次 |
| CI Worker | `workers: 1` | CI 环境单线程执行，避免资源竞争 |
| 超时时间 | `timeout: 20000` | 单个测试最长 20 秒 |
| 截图策略 | `only-on-failure` | 仅失败时截图 |
| 追踪策略 | `on-first-retry` | 首次重试时记录追踪 |

**浏览器覆盖：**

| 项目名 | 设备 | 说明 |
|--------|------|------|
| `chromium` | Desktop Chrome | Chromium 内核浏览器 |
| `firefox` | Desktop Firefox | Firefox 浏览器 |
| `webkit` | Desktop Safari | Safari 内核浏览器 |

**运行方式：**

```bash
npm run test:e2e      # 运行全部 E2E 测试（Chrome + Firefox + Safari）
npm run test:e2e:ui   # Playwright UI 模式运行测试
```

CI 环境下会先执行 `npm run build` 构建生产版本，再通过 `npx serve dist -l 3000` 启动静态服务器供测试使用。

## 测试文件列表

### page-load.spec.js -- 页面加载测试

**用途：** 验证各核心页面能否正常加载和渲染

测试覆盖 7 个页面：

| 页面 | 路由 | 验证内容 |
|------|------|----------|
| 首页 | `/` | 页面标题包含"侗乡医药"，导航栏和首页容器可见 |
| 药用植物 | `/plants` | 植物卡片列表可见 |
| 知识库 | `/knowledge` | 知识卡片列表可见 |
| 传承人 | `/inheritors` | 传承人卡片列表可见 |
| 问答 | `/qa` | 问答卡片/条目可见 |
| 学习资源 | `/resources` | 资源卡片列表可见 |
| 反馈 | `/feedback` | 反馈表单可见 |

首页测试还验证了搜索功能：在植物页面输入"钩藤"进行搜索。

### auth.spec.js -- 用户认证流程测试

**用途：** 验证登录、注册、验证码等认证相关功能

| 测试用例 | 类型 | 验证内容 |
|----------|------|----------|
| 登录页面应可访问 | UI | 登录表单可见 |
| 注册页面应可访问 | UI | 注册页面正常加载 |
| 登录 API 缺少验证码应返回 400 | API | 参数校验生效 |
| 完整登录流程应成功 | API | 验证码接口返回 `captchaKey` |
| 注册 API 缺少验证码应返回 400 | API | 参数校验生效 |
| 验证码 API 应返回图片 | API | 返回 `data:image` 格式的 Base64 图片 |

**设计思路：** 认证测试采用 API 级别测试（`request` 上下文）而非纯 UI 测试，因为验证码图片和表单交互在 E2E 中难以自动化验证，而 API 返回结构可以精确断言。

### navigation.spec.js -- 详情页面导航测试

**用途：** 验证从列表页进入详情页的导航流程

| 测试用例 | 验证内容 |
|----------|----------|
| 植物详情页应可访问 | 点击植物卡片后 URL 包含 `/plants` |
| 知识详情页应可访问 | 点击知识卡片后页面正常跳转 |
| 植物详情 API 应返回正确结构 | 列表取第一条记录的 `id`，请求详情接口验证 `nameCn` 字段 |
| 知识详情 API 应返回正确结构 | 列表取第一条记录的 `id`，请求详情接口验证 `code === 200` |

**设计思路：** 导航测试采用"先获取列表再验证详情"的两步策略，避免硬编码 ID 导致测试数据依赖。

### api.spec.js -- API 接口综合测试

**用途：** 全面验证后端 API 接口的正确性、安全性和一致性

包含 6 个测试分组：

#### 1. API 接口基础验证

验证植物、知识、传承人、问答、资源列表接口及统计、验证码、浏览量递增、反馈提交等接口返回 200 和正确的数据结构。

#### 2. 认证白名单回归测试

验证以下 GET 接口无需登录即可访问（不返回 401）：

```
/api/plants/list, /api/knowledge/list, /api/inheritors/list,
/api/qa/list, /api/resources/list, /api/feedback/stats,
/api/captcha, /api/leaderboard/combined, /api/quiz/questions, /api/metadata/filters
```

白名单 POST 请求（反馈提交、浏览量递增）同样不需要登录。

#### 3. 分页限制回归测试

| 场景 | 验证 |
|------|------|
| `size=9999` | 返回记录数不超过 100 |
| `page=-1` | 实际页码 >= 1 |

#### 4. 安全防护回归测试

| 攻击类型 | Payload 示例 | 验证 |
|----------|-------------|------|
| XSS | `<script>alert(1)</script>` | 状态码 < 500 |
| SQL 注入 | `1 OR 1=1`、`'; DROP TABLE users;--` | 状态码 < 500 |
| 路径遍历 | `../../../etc/passwd` | 状态码不为 200 |

#### 5. 数据一致性测试

| 场景 | 验证 |
|------|------|
| 植物详情与列表数据一致 | 列表首条记录的 `id` 与详情接口返回的 `id` 一致 |
| 反馈统计一致 | `total === pending + resolved` |

#### 6. 响应格式测试

验证所有 API 返回统一的 `{ code, msg, data }` 格式，`code` 为数字类型。

### regression.spec.js -- 回归测试

**用途：** 防止已修复的 Bug 再次出现，覆盖分页边界、历史 Bug、数据完整性和错误处理

#### 分页边界测试

| 场景 | 验证 |
|------|------|
| `size=9999` | 记录数 <= 100 |
| `size=0` | 实际 size >= 1 |
| `page=-1` | 实际 page >= 1 |
| `size=100` | 正常返回 |
| `count=999`（测验题目） | 返回题目数 <= 50 |
| `page=0` | 实际 page >= 1 |
| `size=-5` | 实际 size >= 1 |

#### 历史 Bug 回归测试

验证以下已修复问题不会复现：

- 浏览量递增接口不应返回 401（植物、知识、传承人）
- 反馈提交不应返回 401
- 列表接口 `size` 参数不应返回全量数据
- XSS/SQL 注入搜索不应导致服务器错误
- 评论列表、排行榜、测验题目、验证码、反馈统计、资源下载接口应无需登录

#### 数据完整性测试

| 场景 | 验证 |
|------|------|
| 植物列表记录 | 包含 `id`、`nameCn` 字段 |
| 知识列表记录 | 包含 `id` 字段 |
| 传承人列表记录 | 包含 `id` 字段 |
| 分页元数据 | 包含 `page`、`size`、`total`、`records` 字段 |

#### 错误处理测试

| 场景 | 预期状态码 |
|------|-----------|
| 访问不存在的植物/知识/传承人 | 404 |
| 反馈提交空内容 | 400 |
| 反馈提交缺少类型 | 400 |
| 登录缺少验证码 | 400 |
| 注册缺少验证码 | 400 |


## 测试编写规范

### 两类测试上下文

本项目 E2E 测试使用 Playwright 提供的两种上下文：

- **`page`**：浏览器页面上下文，用于 UI 交互测试（点击、输入、导航等）
- **`request`**：HTTP 请求上下文，用于 API 接口测试（直接发送 HTTP 请求，不经过浏览器）

```javascript
// UI 测试示例
test('页面应可访问', async ({ page }) => {
  await page.goto('/plants')
  await expect(page.locator('.plant-card')).toBeVisible()
})

// API 测试示例
test('接口应返回200', async ({ request }) => {
  const response = await request.get('/api/plants/list')
  expect(response.ok()).toBeTruthy()
})
```

### 容错性设计

测试用例中大量使用 `if (await element.isVisible())` 的条件判断而非硬断言，这是因为：

1. E2E 测试环境数据可能不完整，硬断言容易导致测试不稳定
2. 部分功能依赖登录状态，未登录时 UI 元素可能不显示
3. 关键路径使用严格断言，次要路径使用条件判断

## 代码审查与改进建议

- [稳定性] 部分测试使用 `page.waitForTimeout(2000)` 硬等待，应替换为 `waitForSelector` 或 `waitForResponse` 等显式等待
- [覆盖率] 缺少管理后台页面的 E2E 测试
- [覆盖率] 缺少评论提交、收藏操作等用户交互流程的测试
- [覆盖率] 缺少植物识别游戏和趣味答题的完整流程测试
- [稳定性] `websocket.spec.js` 仅验证连接建立，未验证消息收发和断线重连逻辑
- [安全] `api.spec.js` 的安全测试覆盖了 XSS 和 SQL 注入，但缺少 CSRF 和越权访问测试
