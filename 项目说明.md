## 一、项目概述

### 1.1 项目定位

侗乡医药数字展示平台是一个基于 **Vue 3 + Spring Boot** 的前后端分离毕业设计项目，以"保护、传承、活态传播"为核心理念，通过数字化手段展示三江侗族自治县的传统医药文化。项目聚焦于广西壮族自治区级非物质文化遗产的数字保护与传播，旨在解决侗族医药文化传承面临的三大痛点：

- **传承断层**：老一辈侗医逐渐凋零，年轻一代缺乏接触渠道
- **资料散失**：传统药方、疗法以口传心授为主，缺乏系统化记录
- **传播受限**：侗族医药文化影响力局限于当地，难以触达更广泛受众

### 1.2 核心概念

| 概念 | 说明 |
|------|------|
| 非遗数字化 | 将非物质文化遗产通过数字技术进行采集、存储、展示和传播 |
| 侗族医药 | 侗族人民在长期与疾病斗争中形成的传统医药体系，包含药治、外治、精神治疗三大体系 |
| 传承人 | 经国家/省/市/县认定的非物质文化遗产项目代表性传承人 |
| 药浴 | 侗族最具代表性的外治疗法，将草药煎煮后用于全身或局部浸泡 |
| 钩藤 | 侗族地区道地药材，以贵州从江、广西三江出产为佳，具有息风定惊、清热平肝之功效 |
| 透骨草 | 侗医常用药材，祛风除湿、活血止痛，主治跌打损伤 |
| 九节茶 | 又名草珊瑚，侗族民间常用于清热解毒、祛风通络 |

### 1.4 前后端分离架构

```
前端(Vue 3)：负责页面展示和用户交互 <--JSON数据--> 后端(Spring Boot)：负责业务逻辑和数据存储
```

前端和后端是两个独立的项目，通过 RESTful API 接口（传递 JSON 数据）进行通信。Nginx 作为反向代理，根据请求路径将流量分发到前端静态资源或后端 API。

### 2.2 请求处理流水线

每个 HTTP 请求经过以下处理链：

```
请求进入
  |
  v
[RequestIdFilter] -- 生成16位UUID写入MDC + 响应Header
  |
  v
[XssFilter] -- 检测30+种XSS攻击模式，JSON/表单双模式清洗
  |
  v
[RequestSizeFilter] -- 检查请求体大小（默认10MB上限）
  |
  v
[SaTokenInterceptor] -- 路由拦截：写操作需登录，读操作放行
  |
  v
[RateLimitAspect] -- Redis Lua脚本限流 + 本地令牌桶降级
  |
  v
[LoggingAspect] -- 记录请求开始时间、参数脱敏
  |
  v
[Controller] -- 业务处理
  |
  v
[GlobalExceptionHandler] -- 异常统一捕获，返回R<T>格式
  |
  v
响应返回（含requestId追踪）
```

### 2.3 双层缓存架构

```
请求查询
  |
  v
[Caffeine L1 本地缓存] -- 命中？ --> 直接返回
  |                           maxSize=1000, expireAfterWrite=10min
  | 未命中
  v
[Redis L2 分布式缓存] -- 命中？ --> 回填L1 + 返回
  |                           TTL: 5min~12h（按缓存名配置）
  | 未命中
  v
[MySQL 数据库] -- 查询结果 --> 回填L2 + 回填L1 + 返回

降级策略：Redis不可用时自动降级为Caffeine单层缓存
```

### 2.8 数据流示例：用户查看药用植物

```
1. 用户在浏览器点击"钩藤"卡片
2. Vue Router 跳转到 /plants 页面，触发详情弹窗
3. Vue 组件调用 Axios 发送 GET /api/plants/1 请求
4. Nginx 识别 /api/ 前缀，将请求转发到后端 Spring Boot
5. Sa-Token 拦截器验证该接口是否需要登录（此接口不需要）
6. PlantController 接收请求，调用 PlantService
7. PlantService 先查 Caffeine L1 缓存 --> 未命中则查 Redis L2 缓存 --> 未命中则查 MySQL
8. MySQL 返回植物数据 --> 存入 Redis 缓存 --> 存入 Caffeine 缓存 --> 返回给 Controller
9. 异步更新浏览量和人气值（PopularityAsyncService）
10. 若用户已登录，异步记录浏览历史（BrowseHistoryService）
11. Controller 用 R<Plant> 包装成统一格式：{code:200, msg:"success", data:{...}, requestId:"a1b2c3"}
12. JSON 响应返回浏览器 --> Vue 组件渲染页面 --> 用户看到"钩藤"详情
```

### 2.9 完整登录数据流（全链路）

```
1. 用户输入用户名+密码 --> 前端 sanitizeRequestData 清洗 --> Axios 携带验证码发送 POST /api/user/login
2. Nginx 识别 /api/ --> 转发到后端 Spring Boot
3. RequestIdFilter --> 生成 16 位 UUID 写入 MDC + 响应 Header
4. XssFilter --> 检测请求体中的 XSS 攻击模式（30+ 种），发现则拦截返回 400
5. RequestSizeFilter --> 检查请求体大小（默认 10MB 上限）
6. SaTokenConfig 拦截器 --> /api/user/login 在 excludePathPatterns 中（放行）
7. RateLimitAspect --> 检查登录接口限流（5 次/周期），超限返回 429
8. LoggingAspect --> 记录请求开始时间
9. UserController.login() --> 参数校验 --> CaptchaService 验证验证码（Redis 5分钟过期）
10. UserService.login() --> BCrypt 比对密码 --> StpUtil.login() 生成 Sa-Token
11. 返回 R.ok(tokenValue, userId, username, role) --> LoggingAspect 记录耗时
12. 前端收到 token --> 存入 Pinia userStore + localStorage --> 后续请求自动携带 Authorization Header
```

### 2.12 后端启动类设计

`DongMedicineBackendApplication.java` 关键设计：

```java
@SpringBootApplication(exclude = {
    org.springframework.boot.autoconfigure.amqp.RabbitAutoConfiguration.class
})
@EnableCaching
public class DongMedicineBackendApplication {
    public static void main(String[] args) {
        // 1. 先加载 .env 文件（dotenv-java）
        Dotenv dotenv = Dotenv.configure()
            .directory("./")
            .ignoreIfMissing()
            .load();
        // 2. 设置系统属性供 Spring 读取
        dotenv.entries().forEach(entry ->
            System.setProperty(entry.getKey(), entry.getValue()));
        // 3. 启动 Spring Boot
        SpringApplication.run(DongMedicineBackendApplication.class, args);
    }
}
```

设计要点：
- **排除 RabbitMQ 自动配置**：通过 `exclude` 避免未配置 RabbitMQ 时启动失败，改为条件加载
- **Dotenv 优先加载**：在 Spring 上下文创建前加载 `.env`，确保所有配置属性可用
- **@EnableCaching**：显式启用缓存，配合 CacheConfig 实现双层缓存

### 4.7 常见安装问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|

### 5.1 后端三层架构

后端严格遵循 **Controller -> Service -> Mapper** 三层架构：

```
+------------------------------------------------------------------+
| Controller 层 - 接收 HTTP 请求                                     |
| 注解: @RestController + @RequestMapping + @Validated              |
| 注入: @RequiredArgsConstructor + private final Service             |
| 返回: R<T> 统一响应格式                                            |
+------------------------------+-----------------------------------+
                               | 调用 service.xxx()
                               v
+------------------------------------------------------------------+
| Service 层 - 业务逻辑                                              |
| 接口: extends IService<Entity>                                     |
| 实现: @Service + extends ServiceImpl<Mapper, Entity>               |
| 缓存: @Cacheable + @CacheEvict                                     |
| 事务: @Transactional(rollbackFor = Exception.class)               |
+------------------------------+-----------------------------------+
                               | 调用 mapper / baseMapper
                               v
+------------------------------------------------------------------+
| Mapper 层 - 数据访问                                               |
| 接口: @Mapper + extends BaseMapper<Entity>                        |
| 自动方法: insert, deleteById, updateById, selectById 等            |
| 自定义: @Select / @Update 注解                                     |
+------------------------------+-----------------------------------+
                               v
                          MySQL 8.0
```

**统一响应格式 R\<T\>**：

```json
{
  "code": 200,
  "msg": "success",
  "data": { ... },
  "requestId": "a1b2c3d4e5f67890"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| code | int | 状态码：200成功, 400参数错误, 401未认证, 403无权限, 404不存在, 500服务器错误 |
| msg | String | 消息描述 |
| data | T | 业务数据（泛型） |
| requestId | String | 请求追踪ID（16位UUID），用于日志排查 |

**分页响应格式**：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "records": [...],
    "total": 100,
    "current": 1,
    "size": 12,
    "pages": 9
  },
  "requestId": "..."
}
```

### 5.7 MQ 层详解（5 个 Producer + 5 个 Consumer）

| 业务场景 | 生产者 | 消费者 | Exchange | Queue | 目标存储 |
|---------|--------|--------|----------|-------|---------|
| 操作日志记录 | OperationLogProducer | OperationLogConsumer | operation.exchange | operation.queue | MySQL operation_log 表 |
| 用户反馈提交 | FeedbackProducer | FeedbackConsumer | feedback.exchange | feedback.queue | MySQL feedback 表 |
| 文件异步处理 | FileProcessProducer | FileProcessConsumer | file.exchange | file.queue | 文件系统 |
| 浏览/点赞/分享统计 | StatisticsProducer | StatisticsConsumer | statistics.exchange | statistics.queue | Redis counter |
| 系统通知 | NotificationProducer | NotificationConsumer | notification.exchange | notification.queue | Redis List |

RabbitMQ 配置详情：
- 3 个 Exchange（Direct/Topic/DLX）
- 5 个 Queue（均启用 Lazy 模式，减少内存占用）
- 5 个 DLQ（死信队列，处理失败消息）
- Jackson2JsonMessageConverter JSON 序列化
- 指数退避重试（1s/2s/4s，最多 3 次）
- `@ConditionalOnProperty("app.rabbitmq.enabled")` 条件启用
- 不可用时自动降级为同步执行

### 10.1 认证授权

```java
    Recipe getDetail(Integer id);
    void clearCache();
}

// RecipeServiceImpl.java
@Service
@RequiredArgsConstructor
@Slf4j
public class RecipeServiceImpl extends ServiceImpl<RecipeMapper, Recipe> implements RecipeService {

    @Override
    @Cacheable(value = "recipes", key = "'search:' + (#keyword ?: 'all') + ':page:' + #page")
    public Page<Recipe> searchPaged(String keyword, Integer page, Integer size) {
        Page<Recipe> pageParam = PageUtils.getPage(page, size);
        if (keyword != null && !keyword.isBlank()) {
            String escaped = PageUtils.escapeLike(keyword);
            LambdaQueryWrapper<Recipe> wrapper = new LambdaQueryWrapper<Recipe>()
                .like(Recipe::getNameCn, escaped)
                .or().like(Recipe::getIndications, escaped)
                .orderByDesc(Recipe::getPopularity);
            return page(pageParam, wrapper);
        }
        return page(pageParam, new LambdaQueryWrapper<Recipe>().orderByDesc(Recipe::getPopularity));
    }

    @Override
    @Cacheable(value = "recipes", key = "'detail:' + #id")
    public Recipe getDetail(Integer id) {
        return getById(id);
    }

    @Override
    @CacheEvict(value = "recipes", allEntries = true)
    public void clearCache() { }
}
```

## 十三、架构决策记录

### ADR-001: 选择 Sa-Token 而非 Spring Security

| 项目 | 内容 |
|------|------|
| **状态** | 已采纳 |
| **背景** | 项目需要认证授权框架，主流选择为 Spring Security 和 Sa-Token |
| **决策** | 采用 Sa-Token + JWT 模式 |
| **原因** | 1. **学习曲线低**：Sa-Token API 简洁直观，`StpUtil.login(id)` 一行代码完成登录，而 Spring Security 需要大量配置类 2. **功能丰富**：内置踢人下线、同端互斥登录、Token 自动续期、临时 Token 切换身份等 Spring Security 需要大量自定义实现的功能 3. **JWT 集成简单**：`sa-token-jwt` 插件一行配置即可整合 JWT 4. **注解鉴权**：`@SaCheckRole("admin")` 比 Spring Security 的 `@PreAuthorize` 更简洁 5. **轻量级**：Sa-Token 核心 JAR 仅 200KB+，Spring Security 体系庞大 6. **中文文档完善**：官方文档为中文，适合国内开发 |
| **权衡** | Sa-Token 社区规模小于 Spring Security；若团队已熟悉 Spring Security，迁移成本需考虑 |
| **替代方案** | Spring Security + JWT：功能更全面，但配置复杂度高，对毕设项目过于重量级 |

### ADR-002: 选择 MyBatis-Plus 而非 JPA/Hibernate

| 项目 | 内容 |
|------|------|
| **状态** | 已采纳 |
| **背景** | 项目需要 ORM 框架实现数据库操作 |
| **决策** | 采用 MyBatis-Plus 作为 ORM 框架 |
| **原因** | 1. **SQL 可控**：复杂查询直接写 SQL，性能可预期，不像 JPA 的 N+1 问题难以排查 2. **代码生成**：`BaseMapper<T>` 提供通用 CRUD，`ServiceImpl<M,T>` 提供通用 Service 层方法，减少样板代码 3. **分页插件**：内置分页拦截器，一行配置即可使用，JPA 分页需要额外封装 4. **QueryWrapper**：动态条件构造器，类型安全，比 JPA 的 Specification 更直观 5. **全文搜索**：自定义 `@Select` 注解直接编写 MATCH AGAINST 语句，JPA 实现全文搜索非常困难 6. **国内主流**：国内 Spring Boot 项目使用率最高的 ORM 框架，社区活跃，问题易解决 |
| **权衡** | JPA 的自动建表和跨数据库能力更强；MyBatis-Plus 需要手动维护 SQL |
| **替代方案** | Spring Data JPA：自动建表、方法名查询更方便，但复杂查询灵活性差，性能调优困难 |

### ADR-003: 选择 Vue 3 而非 Vue 2

| 项目 | 内容 |
|------|------|
| **状态** | 已采纳 |
| **背景** | 项目前端框架选型，Vue 2 已于 2023 年 12 月 31 日停止维护 |
| **决策** | 采用 Vue 3 + Composition API |
| **原因** | 1. **官方推荐**：Vue 2 已停止维护，Vue 3 是唯一官方推荐版本 2. **Composition API**：逻辑复用更灵活，Composable 模式比 Mixin 更清晰，无命名冲突 3. **性能提升**：Vue 3 重写虚拟 DOM，编译期优化（静态提升/补丁标记），渲染速度提升 30-50% 4. **TypeScript 支持**：Vue 3 从底层重写，TypeScript 支持更完善 5. **生态兼容**：Element Plus、Pinia、Vue Router 4 均已稳定支持 Vue 3 6. **Vite 加持**：Vite 原生支持 Vue 3，开发服务器启动速度比 Webpack 快 10-100 倍 |
| **权衡** | 部分 Vue 2 生态库尚未迁移到 Vue 3；但本项目使用的核心库均已支持 |
| **替代方案** | Vue 2 + Options API：已有项目迁移成本高时考虑，新项目不推荐 |

### ADR-004: 双层缓存架构（Caffeine L1 + Redis L2）

| 项目 | 内容 |
|------|------|
| **状态** | 已采纳 |
| **背景** | 项目需要缓存策略提升读取性能，降低数据库压力 |
| **决策** | 采用 Caffeine（本地缓存）+ Redis（分布式缓存）双层架构 |
| **原因** | 1. **性能分层**：L1 Caffeine 命中率约 60-80%，响应时间 <1ms；L2 Redis 命中率约 90%+，响应时间 <5ms；数据库响应时间 10-100ms 2. **自动降级**：Redis 不可用时自动降级为 Caffeine 单层，系统不宕机 3. **Spring Cache 统一抽象**：`@Cacheable`/`@CacheEvict` 注解透明使用，无需关心底层实现 4. **缓存一致性**：写操作 `@CacheEvict` 同时清除 L1 和 L2，保证数据一致性 5. **TTL 分级**：L1 短 TTL（5-30 分钟），L2 长 TTL（1-4 小时），平衡实时性和命中率 |
| **架构图** | |

```
请求 --> Caffeine L1 (本地内存, <1ms)
              | 命中 --> 返回
              | 未命中 --> Redis L2 (分布式, <5ms)
                              | 命中 --> 写入 L1 --> 返回
                              | 未命中 --> 数据库 --> 写入 L1 + L2 --> 返回
```

| **权衡** | 双层缓存增加了内存使用和一致性维护复杂度；单机部署时 Redis 价值有限 |
| **替代方案** | 纯 Redis 缓存：简单但每次请求都有网络开销；纯 Caffeine 缓存：多实例部署时缓存不一致 |

### ADR-005: 引入 RabbitMQ 消息队列

| 项目 | 内容 |
|------|------|
| **状态** | 已采纳 |
| **背景** | 浏览量/人气值更新等非关键路径操作需要异步化，避免阻塞主请求 |
| **决策** | 采用 RabbitMQ 作为消息中间件 |
| **原因** | 1. **异步解耦**：浏览量更新、人气值计算等操作异步执行，主接口响应时间从 50-100ms 降至 10-20ms 2. **削峰填谷**：高并发时消息排队处理，避免数据库连接池耗尽 3. **可靠投递**：消息确认机制（ACK）+ 持久化，确保消息不丢失 4. **自动降级**：RabbitMQ 不可用时自动切换为同步执行，系统不宕机 5. **延迟队列**：支持延迟消息，可用于定时任务（如定期清理过期数据） |
| **使用场景** | 1. 浏览量/人气值异步更新（PopularityAsyncService） 2. 反馈通知消息推送 3. 数据统计异步计算 |
| **权衡** | 引入 RabbitMQ 增加了运维复杂度；项目规模较小时同步调用也可满足需求 |
| **替代方案** | 1. 直接同步更新：简单但影响接口响应时间 2. Redis Stream：轻量但功能不如 RabbitMQ 完善 3. Kafka：过重，适合大数据场景 |

### ADR-006: 前后端分离架构

| 项目 | 内容 |
|------|------|
| **状态** | 已采纳 |
| **背景** | 项目架构选型，可选单体模板引擎或前后端分离 |
| **决策** | 采用 Vue 3 + Spring Boot 前后端分离架构 |
| **原因** | 1. **职责分离**：前端专注 UI/UX，后端专注业务逻辑，团队可并行开发 2. **独立部署**：前端静态资源 CDN 加速，后端 API 独立扩缩容 3. **用户体验**：SPA 单页应用，页面切换无刷新，交互流畅 4. **接口复用**：RESTful API 可同时服务 Web 端、移动端、小程序 5. **技术栈独立**：前端可独立升级 Vue/Vite 版本，后端可独立升级 Spring Boot 版本 |
| **权衡** | 前后端分离增加了跨域处理、接口文档维护、部署配置等复杂度 |
| **替代方案** | Thymeleaf/JSP 模板渲染：简单直接，但交互体验差，不利于前后端分工 |
