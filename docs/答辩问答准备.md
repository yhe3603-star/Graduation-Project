# 答辩问答准备（5分钟）

## 一、架构与技术选型类

### Q1: 为什么选择前后端分离架构而不是传统的服务端渲染？

**答**：前后端分离有三个核心优势：第一，**独立部署**——前端静态资源通过Nginx直接返回，后端只处理API请求，职责清晰；第二，**开发效率**——前后端可以并行开发，通过API文档对齐接口；第三，**扩展性**——前端可以轻松接入小程序、APP等多端，后端也可以为第三方提供API。对于这个需要多媒体展示和AI对话的平台，前后端分离能更好地处理WebSocket长连接和流式渲染。

### Q2: 为什么用Spring Boot 3而不是其他后端框架？

**答**：Spring Boot 3基于Java 17，支持虚拟线程等新特性，自动配置机制减少样板代码。MyBatis-Plus提供了强大的条件构造器和分页插件，适合动态查询场景。Sa-Token相比Spring Security更轻量，JWT模式天然支持前后端分离的无状态认证。这三者组合是目前Java生态最成熟的全栈方案。

### Q3: 为什么用MyBatis-Plus而不用JPA？

**答**：MyBatis-Plus的LambdaQueryWrapper可以链式构建动态查询条件，非常适合本项目中"关键词 + 分类 + 疗法"这类多条件可选筛选场景。相比JPA的JPQL，MyBatis直接写SQL更直观，也更容易优化复杂查询。Plus还内置了分页插件和自动填充，减少了重复代码。

## 二、缓存相关

### Q4: 为什么需要两级缓存？只用Redis不行吗？

**答**：只用Redis有两个问题：第一，**网络开销**——每次读取都要走网络IO，对于高频访问的热门数据（如首页展示），网络延迟会累积；第二，**单点风险**——Redis宕机时系统完全不可用。两级缓存中，Caffeine作为L1层处理热数据的微秒级访问，Redis作为L2层保证跨实例的数据一致性。Redis不可用时自动降级为仅Caffeine，保证系统可用性。

### Q5: 缓存一致性怎么保证？数据更新后缓存怎么处理？

**答**：采用**Cache Evict策略**——管理员修改内容后，Service层方法通过`@CacheEvict`注解主动驱逐对应缓存区域。例如修改植物信息后驱逐"plants"缓存，下次读取时重新从数据库加载并回填两级缓存。对于搜索结果这类短TTL缓存（5分钟），采用自然过期策略，不需要主动驱逐。

### Q6: Caffeine缓存满了怎么办？

**答**：Caffeine配置了maximumSize=1000条目的上限和LRU淘汰策略。当缓存满时，最久未访问的条目会被自动淘汰。同时设置了写入后10分钟过期和访问后30分钟过期的双重过期机制，确保不会无限增长。

## 三、认证与安全类

### Q7: Sa-Token的JWT模式和传统Session有什么区别？

**答**：传统Session需要服务端保存会话状态，JWT模式下令牌本身包含用户信息（通过签名防篡改），服务端无需存储Session。本项目使用Sa-Token的UUID令牌+HttpOnly Cookie认证模式：登录成功后由后端自动设置HttpOnly、Secure、SameSite=Lax的安全Cookie，前端不再将令牌存入localStorage（防止XSS窃取），浏览器自动携带Cookie进行认证。同时实现了账户锁定机制（LoginAttemptService），同一用户名连续5次登录失败后锁定15分钟，有效防止暴力破解。Sa-Token内部使用Redis存储令牌-会话映射，对客户端来说是无状态的Cookie认证。

### Q8: 接口限流是怎么实现的？

**答**：使用自定义`@RateLimit`注解 + AOP切面实现。限流算法采用**Redis滑动窗口**——通过Lua脚本原子执行INCR + EXPIRE，窗口大小1秒。Redis不可用时自动降级为**本地令牌桶**——容量100令牌，每秒补充1个。限流粒度是**每个用户每个接口**独立计数，例如登录接口限制某用户5次/秒，不影响其他用户。

### Q9: XSS防护的实现原理是什么？

**答**：三层防护：

1. **前端请求拦截器**：Axios拦截器对所有请求体递归扫描字符串值，匹配30种XSS模式（script标签、事件处理器、javascript协议等）和6种SQL注入模式，检测到则清洗。
2. **后端XssFilter**：基于Jakarta Servlet Filter，对application/json请求使用装饰器模式包装RequestInputStream，对multipart请求递归扫描参数值，使用Jsoup白名单清理HTML标签。
3. **输出层**：前端使用DOMPurify对所有v-html渲染内容进行安全过滤。

## 四、WebSocket与AI对话类

### Q10: 为什么用WebSocket而不是SSE或HTTP轮询？

**答**：核心原因是**需要双向通信**。AI生成过程中用户可能点击"停止"按钮，客户端需要发送`{type: "stop"}`消息给服务端，服务端调用`Disposable.dispose()`取消正在进行的SSE流。SSE是单向的（服务端到客户端），无法实现这个功能。HTTP轮询则延迟太高，无法实现逐token实时推送的打字机效果。

### Q11: WebSocket连接断开后怎么办？

**答**：前端实现了**指数退避重连**机制——最多重试5次，基础延迟3秒，最大延迟30秒。每次重连间隔翻倍。服务端在`afterConnectionClosed`回调中清理会话Map和活跃订阅，并将Redis中的聊天记录刷入MySQL持久化。用户刷新页面后会自动重新建立连接并加载历史会话列表。

### Q12: AI对话的上下文是怎么管理的？

**答**：每次发送消息时，前端将当前会话的**最近10条消息**作为上下文一并发送。服务端在构建DeepSeek API请求时，将系统提示词 + 历史消息 + 当前用户消息组装为messages数组。会话历史在WebSocket会话期间缓存在Redis中，连接断开时批量写入MySQL的chat_history表。前端通过useChatSessions composable管理会话列表，支持切换和删除会话。

## 五、数据库与性能类

### Q13: 中文全文搜索是怎么实现的？

**答**：MySQL 8.0的FULLTEXT索引默认使用ngram解析器处理中文。创建索引时指定`WITH PARSER ngram`，例如`ft_plants_search FULLTEXT(name_cn, name_dong, efficacy, story) WITH PARSER ngram`。查询时使用`MATCH(...) AGAINST(... IN BOOLEAN MODE)`语法。ngram会将中文文本按配置的token大小（默认2）切分为词组进行索引，虽然不如专业分词引擎精准，但对于本项目的数据量完全够用。

### Q14: 多态关联（target_type + target_id）是怎么设计的？

**答**：评论、收藏、浏览历史三张表都使用`target_type`（枚举值：plant/knowledge/inheritor/resource/qa）+ `target_id`（整数ID）的组合来指向任意内容类型。这样一条评论可以关联到植物、知识或传承人，不需要为每种内容类型创建单独的评论表。查询时通过`WHERE target_type = ? AND target_id = ?`精确定位。在Java侧使用枚举类`TargetType`保证类型安全。favorites表还设置了`(user_id, target_type, target_id)`唯一约束防止重复收藏。

### Q15: 性能测试结果为什么这么快？实际部署会一样吗？

**答**：性能测试使用H2内存数据库运行，没有网络IO和磁盘IO开销，所以数据非常快（平均6~18ms）。实际部署使用MySQL，加上网络延迟和并发负载，响应时间会更长。但测试的核心价值是**验证代码逻辑的正确性和并发安全性**——50并发零错误证明了连接池、线程安全和异常处理没有问题。论文中的阈值（≤0.5秒）是基于实际MySQL环境的保守估计。

## 六、项目管理与工程实践类

### Q16: 项目是怎么做版本控制的？

**答**：使用Git进行版本控制，GitHub作为远程仓库。项目共200余次commit，遵循约定式提交规范（Conventional Commits），如`feat:`、`fix:`、`perf:`等前缀。通过GitHub Actions实现CI/CD：push触发测试 → 构建Docker镜像 → 推送到GHCR → SSH部署到服务器。支持自动回滚机制。

### Q17: Docker Compose部署的8个服务是怎么通信的？

**答**：所有服务通过自定义bridge网络`dong-medicine-network`通信，使用服务名作为主机名（如`mysql`、`redis`）。只有4个端口暴露到宿主机：前端80端口、MySQL 3307端口、Grafana 3001端口、RabbitMQ管理界面15672端口。后端8080端口和Redis 6379端口只在容器网络内部可达，不对外暴露，提高安全性。

## 七、可能会被追问的"刁钻"问题

### Q18: 62种药用植物的数据从哪来的？准确吗？

**答**：数据主要来源于三个渠道：三江县文旅局官网的非遗专栏、《广西壮族自治区非物质文化遗产名录（2025版）》、以及传承人工坊公开访谈视频中的药方信息。每条数据都经过交叉核验，但作为本科毕设，数据的权威性有限，平台定位是展示和传播，而非医学参考。论文中也明确指出"数据体量有限"是不足之一。

### Q19: 如果并发量更大（比如1000并发），架构需要怎么改？

**答**：当前架构有几个瓶颈：HikariCP连接池15个连接、Tomcat默认200线程、单实例部署。要支持1000并发需要：第一，数据库连接池扩大 + 读写分离；第二，Spring Boot多实例部署 + Nginx负载均衡；第三，Redis集群模式替代单机；第四，静态资源上CDN；第五，引入限流熔断组件如Sentinel。但本项目定位是非遗展示平台，预期并发量不大，当前架构已经足够。

### Q20: 你觉得自己项目最大的技术难点是什么？

**答**：最大的技术难点是**AI对话的流式通信链路**。从WebSocket连接建立、消息路由、到WebClient流式调用DeepSeek API、SSE逐token解析、再到通过WebSocket推送给前端的Marked渲染，整条链路涉及异步编程、响应式流、线程池管理和连接生命周期管理。特别是"停止生成"功能需要在任意时刻取消Reactor订阅，同时保证聊天记录的完整性。
